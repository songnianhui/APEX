# APEX_Filter — Session-Based Electronic Structure Screening for Transition Metal Clusters

`APEX_Filter` is the maintained downstream screening package in the APEX repository. It starts from `APEX_CAS` outputs (`cas_data.h5`, `FCIDUMP`, structure metadata, optional `cluster_info.yaml`) and drives a session-based active-space workflow:

`load -> enumerate -> uhf -> ccsd -> ccsd-t -> ccsdt -> dmrg-basis -> dmrg -> extrapolate -> report`

The repository still contains a higher-order branch:

`ccsdt -> fno-uccsdtq -> cc-composite`

This package is designed around two principles:

- Keep the Hamiltonian semantics consistent: `UHF / UCC / HAST-UCC / DMRG` all operate on the same active-space model defined by `FCIDUMP`.
- Keep the screening workflow human-steerable: every major step writes machine-readable summaries and spreadsheet-friendly selection files so the user can decide which configurations enter the next stage.

For literature-aligned workflows, `APEX_Filter` now also distinguishes:

- `symmetry_group`: full-cluster point group label from `APEX_CAS` metadata
- `metal_framework_symmetry`: metal-skeleton-only symmetry hint
- `reduction_symmetry`: the downstream `Cn`-style label actually used for family grouping when appropriate
- `family_scheme`: the convention used to define families for a specific cluster class

For Chan-style reproduction, the default is to keep `config_reduction_mode: "none"`, i.e. preserve all literature configurations after family labeling instead of pruning them heuristically.

The session bootstrap file `filter_settings.yaml` is now intentionally narrow in scope:

- it defines the `APEX_CAS` case location and provenance needed by `step1 load`
- it does not carry the downstream screening policy

From `step2 enumerate` onward, the numerical and selection controls live in the
session-local `filter_session/method_controls.yaml`.

For the maintained mainline, `step3-6` observables are compute-only outputs.
Benchmark comparisons remain available as standalone validation utilities, but
they are no longer injected into production session artifacts.

For the closed V1.0.0 workflow, the maintained mainline stops at:

`report`

## Enumeration Vocabulary

`APEX_Filter` now reports enumeration with a fixed layer vocabulary so different
systems are not compared using overloaded `config` counts:

- `raw spin patterns`: site-labeled collinear broken-symmetry sign patterns
- `spin families`: symmetry- or workflow-grouped families
- `spin x oxidation guesses`: unique `(spin pattern, oxidation assignment)` pairs
- `spin x oxidation x d guesses`: unique fully expanded electronic guesses
- `total configs (saved)`: the number kept after optional config reduction

Reference counts used by the current workflow vocabulary:

| System | Raw spin patterns | Spin families | Spin x oxidation | Spin x oxidation x d | Total configs (saved) |
|---|---:|---:|---:|---:|---:|
| `Fe2S2` | 2 | 2 | 2 | 2 | 2 |
| `Fe4S4(SCH3)4` | 6 | 3 | 24 | 24 | 24 |
| `Fe4S4H4` | 6 | 3 | 24 | 600 | 600 |
| `FeMo-co` LLDUC | 35 | 10 | 630 | 78750 | 78750 |

For Chan-style reproduction, the important point is that `Fe4S4 = 24` and
`FeMo-co = 78750` are *not* the same layer. `APEX_Filter` now prints the whole
layer stack explicitly during `step2 enumerate`.

## Quick Example — Session Workflow

`APEX_Filter` does not start from a raw structure alone. It expects an `APEX_CAS` case directory that already contains:

- SCF checkpoint/state
- orbital HDF5 state
- FCIDUMP

Example case layout:

```text
examples/fe4s4h4/
├── inputs/
│   ├── fe4s4h4.xyz
│   ├── fe4s4h4_cas_settings.yaml
│   ├── fe4s4h4_filter_settings.yaml
│   └── fe4s4h4_cluster_info.yaml   ← optional but recommended
└── outputs/                     ← generated by apex-cas scf / buildcas / fcidump
```

### Step 0: Prepare `APEX_CAS` outputs

Typical commands:

```bash
apex-cas prepare examples/fe4s4h4/inputs/fe4s4h4.xyz \
  --case-dir examples/fe4s4h4 \
  --cas-settings examples/fe4s4h4/inputs/fe4s4h4_cas_settings.yaml

apex-cas scf examples/fe4s4h4/inputs/fe4s4h4.xyz \
  --case-dir examples/fe4s4h4 \
  --cas-settings examples/fe4s4h4/inputs/fe4s4h4_cas_settings.yaml

apex-cas buildcas examples/fe4s4h4/inputs/fe4s4h4.xyz \
  --case-dir examples/fe4s4h4 \
  --cas-settings examples/fe4s4h4/inputs/fe4s4h4_cas_settings.yaml

apex-cas fcidump --case-dir examples/fe4s4h4
```

### Step 1: Create a filter config

Start from the shared template:

```bash
cp shared/config/filter_settings_template.yaml examples/fe4s4h4/inputs/fe4s4h4_filter_settings.yaml
```

Minimal useful fields:

```yaml
apex_cas_case_dir: examples/fe4s4h4
structure_path: inputs/fe4s4h4.xyz
cluster_info_path: inputs/fe4s4h4_cluster_info.yaml
fcidump_path: outputs/fcidump/FCIDUMP.*
```

Paths are interpreted relative to `apex_cas_case_dir`. `FCIDUMP.*` is resolved
to the real FCIDUMP file while excluding `.ecore` sidecars.

For runs driven by the CAS YAML, also keep:

```yaml
benchmark_profile: ""
family_scheme: ""
config_reduction_mode: "none"
```

### Step 2: Start a session

```bash
apex-filter load \
  --config examples/fe4s4h4/inputs/fe4s4h4_filter_settings.yaml \
  --session examples/fe4s4h4/filter_session
```

`step1_load/` now records:

- the resolved FCIDUMP path
- the effective `APEX_CAS` settings inherited from sidecar JSONs
- cluster annotations from `cluster_info.yaml` when provided
- active-space arrays re-sliced to the final `selection.txt` / FCIDUMP choice

`filter_session/method_controls.yaml` is also created automatically. This file
is the session-local control surface for later method parameters (`enumerate`,
`uhf`, `ccsd`, `ccsd_t`, `ccsdt`, `dmrg_basis`, `dmrg`, `fno_uccsdtq`).

The intended authority split is:

- `filter_settings.yaml` controls only `step1 load`
- `method_controls.yaml` controls numerical behavior from `step2 enumerate` onward

For a complete `Fe2S2(SCH3)4^{2-}` guide that starts from the maintained case
layout and walks through every maintained step, see:

- [../docs/example.md](/Users/snh/Projects/APEX/docs/example.md)
- [../examples/fe2s2/README_step_by_step.md](/Users/snh/Projects/APEX/examples/fe2s2/README_step_by_step.md)

### Step 3: Run the screening chain

```bash
apex-filter enumerate --session examples/fe4s4h4/filter_session
apex-filter uhf       --session examples/fe4s4h4/filter_session --pick all
apex-filter ccsd      --session examples/fe4s4h4/filter_session --pick "top-per-family 4"
apex-filter ccsd-t    --session examples/fe4s4h4/filter_session --pick "top-per-family 2"
apex-filter ccsdt     --session examples/fe4s4h4/filter_session --pick "top 4"
apex-filter dmrg-basis --session examples/fe4s4h4/filter_session --pick "top 2"
apex-filter dmrg      --session examples/fe4s4h4/filter_session --pick all
apex-filter extrapolate --session examples/fe4s4h4/filter_session
apex-filter report    --session examples/fe4s4h4/filter_session
```

Optional higher-order branch retained in the tree:

```bash
apex-filter fno-uccsdtq --session examples/fe4s4h4/filter_session --pick "top 2" --freeze-occ 2,4
apex-filter cc-composite --session examples/fe4s4h4/filter_session
apex-filter report       --session examples/fe4s4h4/filter_session
```

## Selection Workflow

Every major step writes standardized **CSV-first** selection artifacts into the
corresponding session step directory:

- `selection_candidates.csv`
- `selection_worklist.csv`
- `selection_guide.md`

`selection_worklist.csv` is the main human-editable handoff file. It includes
header comments explaining the file role and, when available, the step energy
used for downstream screening.

Edit `selection_worklist.csv` and mark the `keep` column with any of:

- `1`
- `true`
- `yes`
- `keep`
- `select`
- `include`
- `run`

Then run:

```bash
apex-filter <next-step> --session <dir> --pick "file /path/to/selection_worklist.csv"
```

Notes:

- Worklists are usually pre-filled with `keep=1` when the intended default is
  "send all survivors to the next step".
- `picked_configs.json` is still written, but only as an internal provenance
  record of what the step actually consumed. It is not the primary user-facing
  selection surface.

## CLI Commands

### `apex-filter load`

Load `APEX_CAS` outputs and create a session.

```bash
apex-filter load --config filter_settings.yaml --session <dir>
```

`filter_settings.yaml` is the step-1 bootstrap file. It sets the case directory,
structure metadata, and `FCIDUMP` provenance. Step-2-and-later controls are
kept in `filter_session/method_controls.yaml`.

### `apex-filter enumerate`

Enumerate spin isomers and electronic configurations.

```bash
apex-filter enumerate --session <dir> [--max-configs N] [--forced-oxidation JSON]
```

The enumeration policy is now also controlled from
`filter_session/method_controls.yaml` through the `enumerate` block. The CLI
flags remain available for direct overrides when needed.

### `apex-filter uhf`

Run active-space broken-symmetry UHF on the `FCIDUMP` Hamiltonian.

```bash
apex-filter uhf --session <dir> --pick all
```

Before energies exist, only:

- `all`
- `labels ...`
- `file ...`

are supported.

### `apex-filter ccsd`

Run active-space `UCCSD` from the saved step-3 reference.

### `apex-filter ccsd-t`

Run active-space `UCCSD(T)`.

### `apex-filter ccsdt`

Run active-space `CCSDT` through `HAST-UCC`.

### `apex-filter dmrg-basis`

Prepare the unrestricted DMRG orbital basis:

- `UCCSD natural orbitals`
- split localization
- alpha/beta pairing
- DMRG-specific ordering

`step7` now also writes routine basis QA files:

- `dmrg_basis_qc.json`
- `dmrg_basis_qc.csv`

These are diagnostics for method quality, not state-selection criteria.

### `apex-filter dmrg`

Run active-space DMRG from the saved step-3 state plus the step-7 orbital basis.

Important runtime note:

- `dmrg.backend: pyscf_dmrgci_sz` is the maintained DMRG route for the Fe2S2 oxidized mainline workflow
- CLI overrides such as `--bond-dims 2000,2400` are supported for targeted reruns without editing the session file

### `apex-filter extrapolate`

Fit finite-`M` DMRG energies to an infinite-bond-dimension estimate.

### `apex-filter report`

Build the final ranked cross-method summary after `step9 extrapolate`.

### `apex-filter dmrg`

Run active-space DMRG on selected prepared bases.

The `dmrg` step now:

- reads session-local controls from `filter_session/method_controls.yaml`
- preserves a per-point raw solver log (`*_dmrg.log`)
- records `converged=True/False`, `energy`, and `wall_time_s` in the summary
- preserves the last available energy even when a point is not fully converged

The same session-local control file also governs `enumerate`, so the selection
policy and the numerical backend settings live in one place.

### `apex-filter extrapolate`

Extrapolate DMRG energies to infinite bond dimension.

### `apex-filter report`

Build the final multi-method summary.

Current ranking priority:

- `CC_composite+DMRG_consensus`
- `CC_composite`
- `CCSDT+DMRG_consensus`
- `DMRG_extrapolated`
- `CCSDT`
- `CCSD(T)`

### `apex-filter fno-uccsdtq`

Run the current higher-order branch. Important: the present implementation is explicitly:

`spin-resolved occupied-NO freeze (retain all virtual orbitals)`

It is not yet a full virtual-threshold FNO extrapolation workflow.

### `apex-filter cc-composite`

Compute:

`E_CCSDT(full) + [E_CCSDTQ(FNO) - E_CCSDT(FNO)]`

and feed the result back into the final report layer.

## Session Layout

Example session directory:

```text
filter_session/
├── session.json
├── method_controls.yaml
├── step1_load/
├── step2_enumerate/
├── step3_uhf/
├── step4_ccsd/
├── step5_ccsd_t/
├── step6_ccsdt/
├── step7_dmrg_basis/
├── step8_dmrg/
├── step9_extrapolate/
├── step10_report/
├── step11_fno_uccsdtq/
└── step12_cc_composite/
```

Each step persists:

- picked labels
- result summaries
- selection artifacts
- backend-specific result files (`npz`, scratch outputs, etc.)

File conventions:

- JSON summaries include `_file_role` / `_comment` metadata fields.
- CSV selection files include `#` header comments explaining their role.
- `picked_configs.json` is retained as provenance; manual selection review is
  done through `selection_worklist.csv`.

This allows the workflow to resume step-by-step without rerunning upstream stages.
